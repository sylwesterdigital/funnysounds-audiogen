# ABOUT

## What this project is

This project is a local sound-generation console built around Meta AudioGen / AudioCraft and wrapped in a small Flask web app.

The goal is simple:

- type a sound prompt
- queue many generations
- let the machine grind through them
- preview the results in one browser page
- reuse the WAVs in an editing or ffmpeg workflow

It is not a polished SaaS product, and it does not pretend to be one. It is a practical local tool for generating weird, fast, disposable, and sometimes surprisingly useful sound effects.

## Why it exists

The project exists because the normal “generate one thing from a script, wait, rerun, wait again” loop is miserable when experimenting with lots of prompts.

The app turns that into a usable local workflow:

- queue several prompts
- leave the page open
- monitor the queue
- come back to a pile of WAV files
- audition them directly in the browser

In other words, it is less about model research and more about turning a raw model into a tolerable creative tool.

## What the app actually does

At runtime, the app:

1. serves a single-page local UI
2. accepts prompt + duration + optional filename prefix
3. stores jobs in an in-memory queue
4. uses one background worker to process jobs sequentially
5. loads AudioGen once and reuses the model
6. writes generated WAV files into `outputs/`
7. exposes generated files back to the UI for playback

## What it does not do

Right now it does **not**:

- run jobs in parallel
- offer cancellation of a running job
- persist queue state across restarts
- provide exact model-level progress
- promise fast generation on macOS
- act as a production-grade inference service

## The reality of this setup

This repo is built around a setup that is functional, but quirky.

It is not the clean official “GPU Linux box, supported deps, xformers on, everything easy” story.

It is the practical “make it work locally on macOS” story.

That means this project is partly an app and partly a documented compatibility workaround.

## Why the queue matters

The queue is the most important user-facing feature.

Without the queue, using a large local audio model for sound-design exploration is tedious. With the queue, the app becomes useful:

- drop ten prompts
- wait
- preview finished sounds
- keep the good ones
- ignore the junk

Even with only one worker, queueing is still a major quality-of-life improvement over manually rerunning scripts.

## How the audio is produced

The sounds are generated by model inference.

The app does not search a stock library. It does not “dig up” prerecorded hidden files from the model package. It runs a generative model that predicts audio tokens and decodes them into waveform audio.

That does not mean every sound is good.
It means the sound is synthesized by the model at inference time.

## Why it feels slow

The slowness comes from several layers at once:

- the model is large
- the setup is local
- the current macOS path disables some faster attention machinery
- autoregressive audio generation is expensive
- longer requested durations multiply the cost

So the app is good for queueing many prompts, but it is not a “press button, instant cartoon SFX” tool.

## Why the UI is intentionally simple

The interface is deliberately straightforward:

- prompt field
- duration
- filename prefix
- queue state
- generated file list
- audio players

That keeps the browser side stable and avoids dragging in unnecessary frontend complexity for a local tool.

## Who this is for

This repo is for someone who wants:

- local sound generation
- a queue
- browser playback
- easy reuse in editing pipelines
- a setup they can tinker with

It is especially useful for:

- experimental video editing
- meme / surreal / absurd sound design
- batch prompt exploration
- quick one-off effects
- ffmpeg-based workflows

## Project philosophy

This repo values:

- honesty over pretending the install is clean
- usefulness over purity
- local control over hosted black boxes
- queue-based iteration over one-shot generation
- simple UI over fancy frontend abstractions

## What “done” means here

For this project, “done” does not mean perfect.

It means:

- the app starts
- the model loads
- prompts can be queued
- WAVs get generated
- the files show up in the UI
- they can be played and reused

That is the bar.

## Current limits worth remembering

- queue is sequential, not parallel
- generation can be slow
- first run downloads large model files
- package compatibility is fragile
- the working setup is pinned and patched
- restarting recklessly can cost time because of model reloads

## The important mental model

Think of this repo as a **local generation workstation**, not as a polished consumer app.

It sits between:

- a raw Python script
- and a more complete sound-design tool

That middle ground is exactly why it is useful.

## Suggested next directions

If this repo keeps evolving, the most valuable upgrades are:

- cancel job
- queue reordering
- delete output from UI
- play newest sound automatically
- waveform thumbnails
- ffmpeg export helpers
- metadata log for prompts and outputs
- optional smaller/faster local models
- persistent job history

## Summary

This project is a local Flask front-end over AudioGen that turns text prompts into queued sound-generation jobs and browser-playable WAV outputs.

Its value is not that it is pretty.
Its value is that it makes a heavy local model usable enough to iterate with.
