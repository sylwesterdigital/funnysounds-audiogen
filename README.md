# AudioGen Local Console

A local Flask web app for generating short sound effects from text prompts with Meta AudioGen / AudioCraft, reviewing a queue of jobs, and previewing the generated WAV files in the browser.

This repo is intentionally pragmatic rather than “official”. It documents the setup that actually worked for this project on macOS with Apple Silicon, including the quirks that came up while making AudioCraft usable locally.

## What this app does

The app provides a one-page local UI for:

- writing a text prompt for a sound
- choosing a target duration
- queueing multiple sound-generation jobs
- tracking job state in the browser without full page reloads
- listing generated WAV files
- previewing generated audio in the browser
- reusing the generated files in an ffmpeg workflow

The backend loads the AudioGen model once and keeps it in memory. New requests are put into a queue and processed one by one by a background worker thread.

## What kind of audio it generates

This app uses `facebook/audiogen-medium` through AudioCraft.

Important clarification:

- the sounds are synthesized by the model
- they are not pulled from a fixed library of stock effects
- the model predicts discrete audio tokens and decodes them into waveform audio

That does **not** mean the model is magically guaranteed to be legally or aesthetically perfect. It means the output is generated by inference, not by searching a folder of prerecorded WAV files.

## Current behavior

### Queue behavior

The app **does support multiple queued jobs**, but it currently uses **one worker thread**, so jobs run sequentially, not in parallel.

That means you can drop in many prompts and leave the app generating them, but only one job is actively generating at a time.

### Duration behavior

The current version intentionally removes the hard upper cap on duration in the UI and API.

Practical reality:

- longer durations are much slower
- large jobs can take a long time on macOS
- queueing many long prompts can easily turn into an overnight render

### Progress behavior

The progress bar is **stage-based**, not real token-level generation progress.

Typical stages are:

- loading model
- preparing generation
- generating audio
- writing wav
- finished / failed

This is deliberate. The app does not currently stream fine-grained internal generation progress from AudioCraft to the browser.

## Why the repo uses a patched AudioCraft checkout

The official AudioCraft install path is GPU-oriented and much cleaner on the officially supported stack.

This repo exists in the messier real world:

- Apple Silicon macOS
- FFmpeg 8 from Homebrew
- no usable `xformers`
- no desire to downgrade the whole machine to fit one library
- a need to get a local UI working anyway

Because of that, the repo uses a patched local checkout in `vendor-audiocraft/` instead of pretending that `pip install -U audiocraft` is enough for this setup.

## macOS installation

These instructions are written for a local project folder on macOS, especially Apple Silicon.

### 1) Requirements

You need:

- Python 3.11 already available on your machine
- ffmpeg already available on your machine
- Git
- enough disk space for model downloads
- patience

Large model files will be downloaded on first run.

### 2) Create and activate a virtual environment

```bash
/opt/homebrew/opt/python@3.11/bin/python3.11 -m venv .venv311
source .venv311/bin/activate
python --version
```

### 3) Install core PyTorch packages

```bash
python -m pip install --upgrade pip setuptools wheel
python -m pip install torch==2.1.0 torchvision==0.16.0 torchaudio==2.1.0
```

### 4) Install the supporting Python packages used by the working setup

```bash
python -m pip install \
  flask \
  "numpy<2" \
  "av>=15,<16" \
  soundfile \
  librosa \
  gradio \
  torchmetrics \
  encodec \
  protobuf \
  torchtext==0.16.0 \
  pystoi \
  torchdiffeq \
  demucs \
  flashy \
  hydra_colorlog \
  hydra-core \
  julius \
  num2words \
  pesq \
  sentencepiece \
  spacy==3.7.6 \
  "transformers==4.37.2" \
  "huggingface-hub<1.0"
```

### 5) Clone AudioCraft locally into the repo

```bash
git clone --depth 1 https://github.com/facebookresearch/audiocraft.git vendor-audiocraft
```

### 6) Patch the local AudioCraft checkout

This repo assumes you already have the patched local checkout in place.

The key workarounds are:

- make AudioCraft run without `xformers`
- avoid the old `av==11` pin that breaks against FFmpeg 8
- force `memory_efficient=False`
- use a `transformers` version compatible with `torch==2.1.0`

If your local checkout gets reset, you will need to reapply those changes.

### 7) Install the local patched checkout

```bash
python -m pip install --no-deps -e vendor-audiocraft
```

### 8) Run the app

```bash
python app.py
```

Then open:

```text
http://127.0.0.1:58317
```

## First-run model download

The first successful generation will trigger large model downloads from Hugging Face.

Expect downloads such as:

- compression weights
- tokenizer assets
- `state_dict.bin`
- `model.safetensors`

This is normal.

If downloads appear stuck, check both caches:

```bash
du -sh ~/.cache/huggingface/hub ~/.cache/huggingface/xet 2>/dev/null
```

Why both?

Because Hugging Face may use Xet-based downloads, so not all activity shows up only in `~/.cache/huggingface/hub`.

## How to use the app

1. Start the Flask app.
2. Open the local URL.
3. Enter a sound prompt.
4. Enter a duration.
5. Optionally add a filename prefix.
6. Click **Generate sound**.
7. Watch the queue update.
8. Preview finished WAV files in the **Generated files** panel.
9. Use those WAV files in your editing or ffmpeg pipeline.

## Using the generated files with ffmpeg

Example:

```bash
ffmpeg -y -i input.mp4 -i outputs/test_funny.wav \
-filter_complex "[1:a]adelay=2000|2000,volume=0.9[s];[0:a][s]amix=inputs=2:duration=first[a]" \
-map 0:v -map "[a]" -c:v copy -c:a aac -shortest output.mp4
```

## Quirks and limitations

### 1) It is slow

This setup is slow for good reasons:

- `audiogen-medium` is a large model
- the macOS path is not the ideal official path
- `xformers` is disabled
- memory-efficient attention had to be bypassed
- longer durations mean more autoregressive generation work

### 2) Queue is sequential

The app currently processes one job at a time.

That is useful for “drop many prompts and let it run”, but it is not parallel batch inference.

### 3) The Flask server is for local development

This app runs on Flask’s development server. It is meant for local use, not production deployment.

### 4) The browser progress bar is approximate

The displayed progress reflects job stages, not exact generation completion.

### 5) Large downloads happen on demand

The model is downloaded the first time it is really needed.

### 6) Long prompts do not guarantee better audio

Prompting matters, but audio models can still hallucinate, drift, or produce muddy results.

### 7) No cancellation yet

This version has queueing, but no “cancel running job” control yet.

## Troubleshooting

### `ModuleNotFoundError: No module named 'flask'`

Install Flask into the active venv:

```bash
python -m pip install flask
```

### The browser loads but no files appear

Check that WAVs exist in `outputs/`.

### The app runs but generation is extremely slow

That is expected for this model and this macOS path. Reduce duration and keep the model loaded instead of restarting the script for every sound.

### Hugging Face download seems frozen

Check both caches:

```bash
du -sh ~/.cache/huggingface/hub ~/.cache/huggingface/xet 2>/dev/null
```

### AudioCraft import breaks after updating packages

This repo depends on a patched local checkout and several pinned packages. Random upgrades can easily break the working combination.

## Notes about originality and reuse

Generated sounds are model outputs, not direct file retrievals from the UI or repo.

Still, treat the output like any other generative-media asset:

- listen critically
- trim it
- normalize or compress it if needed
- create variants in ffmpeg or your DAW
- do not assume every output is production-ready
